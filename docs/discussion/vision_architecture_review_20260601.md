## 2. top_view 모델의 근본적 한계 제기

### 문제 인식
- 탑뷰 카메라가 너무 높아 공구가 픽셀상 매우 작게 촬영됨
- 공구함 영역으로 탐지 범위를 제한했음에도 오검출/미검출 빈번
- 객체가 없는 위치에 bbox 생성, 잘 탐지된 위치에 엉뚱한 클래스 레이블

### 핵심 지적
> mAP 0.973은 "이 데이터셋 안에서 얼마나 잘 맞추냐"의 점수일 뿐,  
> 실제 운용에서 신뢰할 수 있는 탐지인지와는 별개다.

흐릿한 데이터로 학습 → 흐릿한 기준으로 평가 → 높은 mAP → 실제로는 못 믿음.  
데이터 품질의 문제가 아니라 **촬영 환경 자체의 구조적 한계**.

### 카메라 높이 변경 불가 이유
탑뷰 카메라는 단순 공구 탐지 외에 향후 **작업자 작업 환경 모니터링**까지 커버할 계획.  
따라서 넓은 시야각 확보를 위해 높이 유지 필수 — 하드웨어 변경 불가.

## 3. ArUco 크롭 방식 논의

**제안**: ArUco 마커로 서랍별 ROI를 정의하고, 크롭된 영역에서만 YOLO 실행.

- 천장 마커 → 공구함 전체 영역 정의
- 서랍 손잡이 마커 → 층별 구분
- 크롭 이미지에서 공구가 프레임을 크게 차지 → 해상도 문제 완화
- 엉뚱한 위치 bbox 원천 차단

**한계**: 기존 v1/v2/v3 데이터가 전체 프레임 기준으로 찍혀 있어 크롭 방식 전환 시 데이터 재수집 필요.

## 4. 그리퍼 카메라 역할 논의

**논점**: 서랍을 열려면 그리퍼가 움직여야 하고, 그 시점에 그리퍼 카메라는 이미 서랍 앞에 위치해 탑뷰보다 내부를 훨씬 잘 볼 수 있다.

> 탑뷰가 "어느 서랍에 뭐가 있는지" 판별하는 역할이 사실상 불필요해지는 것 아닌가?

**역할 분리안**:

| 카메라 | 역할 |
|--------|------|
| 탑뷰 | 서랍 열림/닫힘 상태, staged 공구 감지, FOD, 작업 환경 모니터링(미래) |
| 그리퍼 | 서랍 내부 공구 종류 판별, 픽업 위치/각도 추정, 파지 성공 여부 |

이 경우 탑뷰 YOLO의 **클래스 자체를 재설계**해야 함.  
`multi_tool`, `ratchet_wrench` 같은 공구 종류 클래스 → `drawer_open`, `tool_staged`, `fod` 같은 **상태 클래스**로 전환.

## 5. 근본 질문 — 카메라 비전이 반드시 필요한가

**프로젝트 핵심 목적**: 음성 명령 → 공구 전달 → 반납

카메라가 **반드시** 해야 하는 TASK:
1. 어느 서랍에 어떤 공구가 있는지 파악
2. 공구 픽업 위치/각도 추정

**미결 논점**:
> 1번이 카메라로 해야만 하는 TASK인가?

공구함 슬롯이 고정되어 있고 각 슬롯의 공구가 사전에 정해져 있다면, **DB가 이미 그 정보를 갖고 있다.** 공구 상태(in/out/staged)를 DB가 추적하므로 탑뷰가 실시간으로 판별할 필요가 없을 수 있다.

**카메라 비전이 진짜 필요한 순간**: DB 상태와 실제 물리 상태가 불일치할 때 — 즉 누군가 공구를 손으로 꺼내거나 잘못 반납했을 때의 **reconciliation (S-9 부팅 시 reconciliation 참고)**.
